4 — Icon-Buttons: Design-Entscheidungen (nicht verlieren!)

Diese Regeln sind das Ergebnis vieler Iterationsrunden — beim Neuaufbau 1:1 übernehmen.
Live-Demo: Icon-Buttons (hover um State zu sehen)
Default
Hover (simuliert)
Eigenschaft Wert Begründung / Kontext
Form border-radius: 50% Kreis · konsistent mit FAB und anderen Buttons im Theme
Größe 32px × 32px Trefferzone, optisch kompakt ohne zu klein zu sein
Icon-Größe font-size: 14px Font Awesome · gleiche Größe wie Nav-Link-Text
Abstand gap: 2px zwischen Buttons Sehr eng — Icons wirken als Gruppe, nicht einzeln
Hintergrund (default) transparent Kein Hintergrund im Ruhezustand — Icons schweben im Header
Farbe (default) var(--text-muted) = #888 Zurückhaltend · nicht ablenkend von Inhalten
Farbe (hover) var(--color-primary) = ■ #3d8e12 Grün — nicht Weiß. Designentscheidung: Hover = Markenfarbe. Konsistent mit Nav-Link-Hover.
Hintergrund (hover) var(--bg-elevated) = #2e2e2e Subtiles Kreis-Highlight · verstärkt die Kreis-Form ohne zu dominant zu sein
Transition color 0.15s, background 0.15s Schnell genug um reaktiv zu wirken, nicht abrupt
Focus-Ring outline: none Bewusst entfernt für Header-Icons · visuell störend im engen Icon-Bereich
Padding padding: 0 Alles über width/height geregelt — kein unerwarteter Padding-Effekt
Theme-Toggle Besonderheit overflow: hidden Verhindert Icon-Überlauf bei FA-Icon-Wechsel (moon ↔ sun); JS: syncIcon()

5 — Nav-Links & Header-Bar: Design-Entscheidungen

Auch hier: das Ergebnis expliziter Variantenentscheidungen — insbesondere Hover Variant A (grün, kein Hintergrund).
Live-Demo: Nav-Links (hover für State)
origr3n Tools Tag-Cloud Bildwand Täglich
"Tag-Cloud" = aktive Seite (grün) · hover über andere Links
Eigenschaft Wert Begründung / Kontext
Farbe (default) var(--text-secondary) = #b0b0b0 Nicht text-muted! Nav-Links sind heller als Icon-Buttons (#707070) — stärkere Präsenz, weil primäre Navigation
Farbe (hover / fokus) var(--color-primary) = ■ #3d8e12 Variant A (aktiv gewählt) — grüner Text, kein Hintergrund-Highlight. Variante B (Hintergrund) wurde verworfen.
Hintergrund (hover) transparent Bewusst kein Fill — ruhigeres Erscheinungsbild, nur Farbe signalisiert Interaktivität
Aktive Seite color: var(--color-primary) Identisch mit Hover-State. Kein Unterstrich, kein Bold, kein Hintergrund — Farbe reicht
Font-Size 0.875rem (14px) Gleich wie Icon-Buttons-Fontgröße · kompakt, nicht zu dominant
Padding Desktop 0 0.75rem Vertikal 0 (Höhe über parent height:52px + align-items:center), horizontal 0.75rem Luft
Transition color 0.15s, background 0.15s Identisch mit Icons — einheitliches Reaktionsgefühl im gesamten Header
Weißer Balken (Shaarli) ::after { display: none !important } Shaarli setzt hover::after { background:#fff; height:4px } — explizit unterdrückt, passt nicht ins Design
Header-Bar (#shaarli-menu)
Hintergrund var(--bg-surface) = #2a2a2a Leicht heller als Page-BG (#1e1e1e) — Header hebt sich ab
Unterlinie 1px solid var(--border-color) = #3a3a3a Subtile Trennung zum Content
Schatten 0 2px 8px rgba(0,0,0,0.45) Tiefe unter dem Header, besonders wichtig bei sticky
Position sticky; top: 0; z-index: 100 Header bleibt beim Scrollen sichtbar

7 — Implementierungsstrategie

Schritt-Plan
①  CSS: #origr3n-header als eigener Block — display: flex; align-items: center; height: 52px
②  CSS: Alle bestehenden #shaarli-menu .pure-menu-horizontal Overrides entfernen
③  CSS: Nav-Links mit height: 52px; display: flex; align-items: center — kein Framework nötig
④  Template: Neue IDs/Klassen auf bestehende <ul> und <div> aufsetzen (kein Umbau der Struktur nötig!)
⑤  Mobile bleibt unberührt (separater Block, eigene CSS-Regeln)
⑥  FAB (Shaare) bleibt CSS-only wie bisher
Kein Template-Umbau notwendig — nur neue CSS-Klassen auf bestehende Elemente, alte Overrides entfernen.
